# Reference Planning

Use this skill to design the document system before drafting content. Favor it when the user wants a durable repository rather than a single memo.

## Workflow
1. Define the audience and the decisions the repository must support.
2. Decompose the topic into layers: evidence, reference, analysis, strategy, QA, and maintenance.
3. Choose the folder and file model.
4. Define phased build order and handoffs between phases.
5. Define what each file must answer so downstream strategy does not invent missing inputs.

## Core Rules
- Plan by reader entry point: entity, product or pathway, or strategy question.
- Tie every file to a `decision supported`, not just a topic.
- Push reusable rules into templates and overview files before repeating them in phase plans.
- Separate evidence collection, interpretation, comparison, and recommendation into different layers.
- Make the strategy layer depend on upstream outputs explicitly.

## Output Expectations
- Produce a repository structure that can scale.
- Define file purpose clearly enough that another agent can write the content without guessing.
- Keep phase plans evidence-first and decision-complete.
