# Phasing Patterns

Use phased builds when the topic is evidence-heavy, comparative, or strategy-led.

## Preferred order
1. Program design and templates
2. Source harvest
3. Country or jurisdiction references
4. Pathway or product references
5. Comparative analysis
6. Strategy outputs
7. QA and consolidation
8. Update operations

## Handoff rule
Each phase must create the inputs required by the next phase. If the downstream phase needs buyer logic, proof burden, or disqualifiers, add those upstream before planning strategy.

## Wave pattern
Use wave-based completion when the repo covers many jurisdictions:
- Wave 1: high-value or high-confidence markets
- Wave 2: contextual or lower-confidence markets

## Acceptance criteria pattern
Every phase plan should define:
- required inputs;
- required outputs;
- workflow;
- owner roles;
- QA gate;
- test scenarios;
- what the phase is not allowed to do.
