# Writing Standard

**Last updated:** 2026-04-21  
**Purpose:** Define the writing standard for future rewrites so the repository works for mixed teams, not only specialists.

## Audience Standard
Assume the reader may be:
- commercial / BD,
- compliance,
- operations,
- finance,
- leadership,
- or an expert reviewer.

That means every important file must work on two levels at once:

1. **Easy first read for non-specialists**
2. **Real depth and precision for specialists**

## Core Rules
- Start with the practical conclusion, not the background.
- Explain every important acronym or technical term the first time it appears.
- Add a plain-English explanation before dense tables.
- Add a “what this means” paragraph after dense tables.
- State not only what the rule is, but what changes commercially because of it.
- Keep nuance, but never assume the reader already understands why it matters.
- Reader-bridge sections may be added before the canonical template sections, but the core template order underneath should remain recognisable for QA and maintenance.

## Layer-Specific Rules

### Country reference
- Explain how the market actually works.
- Name the buyer and the access gate clearly.
- Say what people often misunderstand.

### Pathway reference
- Explain what is in scope and what is out of scope.
- Make the route economics and proof burden concrete.
- Show why the same pathway performs differently across markets.

### Market analysis
- Define the comparison logic before ranking anything.
- Explain why the winners win and why the losers lose.
- Include disqualifiers, not only positives.

### Strategy
- State the recommendation up front.
- Explain why now, who buys, what must be true first, and what would make us stop.
- Make “better alternatives” explicit.

## Reader Bridges To Use Often
Use sections like:
- `What This Helps You Decide`
- `Executive Takeaway`
- `How To Read This`
- `What This Table Is Really Saying`
- `What Non-Experts Usually Miss`
- `What This Means For Our Team`

## Anti-Patterns To Avoid
- Analyst shorthand without explanation
- Acronym-heavy openings
- Tables without interpretation
- Restating the source without surfacing the implication
- Treating eligibility as if it automatically means commercial attractiveness

## Quality Check
A file is good enough only if:
- a non-expert can explain the core point after one read,
- an expert still finds the nuance credible,
- and the document clearly tells the reader what matters, why it matters, and what to do with that information.
